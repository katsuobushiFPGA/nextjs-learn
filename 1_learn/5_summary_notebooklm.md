# Next.js における Server コンポーネントと Client コンポーネントの活用ガイド

## エグゼクティブ・サマリー

Next.js（App Router）のアーキテクチャは、Server コンポーネントと Client コンポーネントの使い分けによって、パフォーマンスとインタラクティブ性の最適化を実現している。デフォルトでは、レイアウトやページは Server コンポーネントとして動作し、サーバー側でデータをフェッチしてレンダリング結果をストリーミングする。一方で、ブラウザ API やステート管理が必要な場合は Client コンポーネントを使用する。このハイブリッドモデルを正しく理解し、コンポーネントを適切に配置することで、JavaScript バンドルサイズの削減、初回表示速度（FCP）の向上、およびセキュリティの強化が可能となる。

---

## 1. コンポーネントの選定基準

開発者は、実行環境（サーバーまたはクライアント）の特性に応じて、コンポーネントの種類を選択する必要がある。

### Server コンポーネントを使用すべきケース

- **データフェッチ**: データベースや API などのソースに近い場所でデータを取得する
- **セキュリティ**: API キーやトークンなどの機密情報をクライアントに公開せずに保持する
- **パフォーマンス**: ブラウザに送信する JavaScript 量を削減し、FCP を改善する
- **ストリーミング**: コンテンツを段階的にクライアントへ送信する

### Client コンポーネントを使用すべきケース

- **インタラクティブ機能**: `onClick` や `onChange` などのイベントハンドラーの使用
- **ステート・ライフサイクル**: `useState`、`useReducer`、`useEffect` などの利用
- **ブラウザ API**: `window`、`localStorage`、`navigator.geolocation` などの参照
- **カスタムフック**: ブラウザの状態やライフサイクルに依存するフックの利用

---

## 2. レンダリングの仕組みと動作フロー

Next.js は、サーバーとクライアントの両方でレンダリングを調整する。

### サーバー上での処理

1. React API を使用して、ルートセグメントごとにレンダリングを分割して実行
2. Server コンポーネントを React Server Component Payload（RSC Payload）というコンパクトなバイナリ形式に変換
3. Client コンポーネントのプレースホルダーと、それらに渡される props を RSC Payload に含める

### クライアント上での処理（初回ロード）

1. **HTML**: サーバーで生成された静的な HTML を即座に表示し、非インタラクティブなプレビューを提供する
2. **RSC Payload**: コンポーネントツリーの調整と DOM の更新に使用
3. **JavaScript**: Client コンポーネントをハイドレート（イベントハンドラーの付与）し、アプリケーションをインタラクティブにする

### ナビゲーション時

その後のページ遷移では、RSC Payload はプリフェッチおよびキャッシュされ、サーバー側での HTML レンダリングを介さずにクライアント上で即座に処理される。

## 3. 実装上の重要パターンと最適化

### `'use client'` ディレクティブの運用

- ファイルの最上部に記述することで、Server モジュールグラフと Client モジュールグラフの「境界」を宣言する
- `'use client'` が指定されたファイルからインポートされるすべてのコンポーネントは、自動的にクライアントバンドルに含まれる
- **最適化**: JavaScript バンドルを最小化するため、`'use client'` はツリーの末端、つまりインタラクティブ性が必要な特定のコンポーネントにのみ適用することが推奨される

### コンポーネントのインターリーブ（交互配置）

Client コンポーネントの中に Server コンポーネントを配置する場合、直接インポートすることはできない。これを解決するために `children` 等の props として Server コンポーネントを渡すパターンを用いる。

| 手法 | 内容 |
|---|---|
| スロット形式 | Client コンポーネントに `children` プロパティを持たせ、親（Server コンポーネント）でラップする |
| 実行順序 | サーバー上で先に Server コンポーネントがレンダリングされ、その結果が Client コンポーネントに渡される |

### Context API の利用

- React Context は Server コンポーネントではサポートされない
- Context を使用する場合、Provider 自体を Client コンポーネントとして作成し、Server コンポーネント（例：`layout.js`）で `children` をラップする形で導入する

### サードパーティライブラリの対応

- `'use client'` ディレクティブが含まれていないクライアント専用ライブラリ（`useState` 等を使用するもの）を Server コンポーネントで直接使うとエラーになる
- 解決策として、そのライブラリを独自の Client コンポーネントでラップしてエクスポートし、それを Server コンポーネントから呼び出す

```tsx
// app/carousel.tsx
'use client'
import { Carousel } from 'acme-carousel'
export default Carousel  // ラップして再エクスポート
```

## 4. 環境汚染の防止とセキュリティ

サーバー専用のコードや機密情報が誤ってクライアントに漏洩することを防ぐための機能が提供されている。

- **環境変数**: `NEXT_PUBLIC_` プレフィックスがない変数はクライアントバンドルから除外（空文字列に置換）される
- **`server-only` パッケージ**: サーバー専用のロジックを含むファイルで `import 'server-only'` を実行することで、そのモジュールが Client コンポーネントにインポートされた場合にビルドエラーを発生させる
- **`client-only` パッケージ**: 同様に、`window` オブジェクトへのアクセスなど、クライアント専用のロジックをマークするために使用される

```ts
import 'server-only'  // クライアントから import されたらビルドエラー

export async function getData() {
  const res = await fetch('https://api.example.com', {
    headers: { authorization: process.env.API_KEY },
  })
  return res.json()
}
```

## 5. 補足事項：データのシリアライズ

Server コンポーネントから Client コンポーネントに props を渡す際、そのデータはシリアライズ可能である必要がある。関数やクラスのインスタンスなどを直接渡すことはできない点に注意が必要である。
