# Next.js 16 デプロイメント戦略ブリーフィング

## 概要

デプロイ形態は大きく4つに分類され、選択によって利用可能な機能が決まる。

---

## デプロイオプション比較

| デプロイ先 | 機能サポート | 特徴 |
|---|---|---|
| Node.js サーバー | **全機能** | 標準的なサーバー環境 |
| Docker コンテナ | **全機能** | Kubernetes 等のコンテナオーケストレーターに適する |
| 静的エクスポート | **制限あり** | HTML/CSS/JS として出力。サーバー依存機能は利用不可 |
| プラットフォームアダプター | **プロバイダー固有** | 各クラウドインフラに合わせて最適化 |

---

## 各デプロイ形態の詳細

### 1. Node.js サーバー

- Next.js の全機能をフル活用できる標準的な手法
- 必要に応じてカスタムサーバーへの変更も可能
- **デプロイ手順**：
  1. `package.json` に `build` / `start` スクリプトを設定
  2. `npm run build` でビルド
  3. `npm run start` でサーバー起動
- テンプレート例：Flightcontrol、Railway、Replit

### 2. Docker コンテナ

- Kubernetes やクラウドコンテナサービスへのデプロイに対応
- **ベストプラクティス**：
  - 本番環境へのデプロイには Docker が最適
  - Mac / Windows のローカル開発では `npm run dev` を推奨（仮想化による I/O 低下を回避）
- テンプレート例：DigitalOcean、Fly.io、Google Cloud Run、Render、SST

### 3. 静的エクスポート

- Next.js を静的サイト（SPA）としてビルドし、静的ファイルとして配信
- **ホスティング先**：AWS S3、Nginx、Apache など任意のウェブサーバー
- **制限**：サーバーを必要とする機能は利用不可
- テンプレート例：GitHub Pages

### 4. プラットフォーム固有のアダプター

- 各プロバイダーのインフラ機能を最大限に活用する形態
- **対応プロバイダー**：Vercel、AWS Amplify Hosting、Cloudflare、Netlify、Deno Deploy
- **将来の展望**：Deployment Adapters API の開発が進行中。完了後は独自アダプターの作成が可能になる予定

---

## 選定指針

| 要件 | 推奨デプロイ先 |
|---|---|
| フル機能が必要 | Node.js サーバー / Docker |
| 低コストでスモールスタート | 静的エクスポート |
| 特定プラットフォームに最適化 | プラットフォームアダプター |
| ローカル開発（Mac / Windows） | `npm run dev`（Docker 不使用） |
