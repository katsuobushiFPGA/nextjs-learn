# Next.jsにおけるリンクとナビゲーションの最適化：技術概要ブリーフィング

## エグゼクティブ・サマリー

Next.js（App Router）におけるナビゲーションは、デフォルトでサーバーレンダリングを採用しながらも、プリフェッチ・ストリーミング・クライアントサイド遷移を組み合わせることで、SPAのような高速でレスポンシブなユーザー体験を提供します。

| 要素 | 概要 |
|---|---|
| サーバー主導の基盤 | レイアウトとページはデフォルトでRSCとして動作し、サーバー側でペイロードが生成される |
| バックグラウンド最適化 | `<Link>` の自動プリフェッチにより、クリック前に遷移先のデータを準備する |
| 段階的表示 | `loading.tsx` の活用により、ルート完成前に一部UIを表示して即座にフィードバックを提供する |
| シームレスな遷移 | クライアントサイド遷移により、ページ全体の再読み込みを回避し共有UIの状態を維持する |

---

## ナビゲーションの基本メカニズム

### 1. サーバーレンダリング

LayoutsおよびPagesはデフォルトでReact Server Componentsです。ナビゲーション時にはサーバー側で「Server Component Payload」が生成されクライアントに送信されます。

レンダリングのタイミングには2つの形態があります。

- **プリレンダリング**: ビルド時または再検証（Revalidation）時に実行され、結果はキャッシュされる
- **ダイナミックレンダリング**: リクエスト時に実行される

### 2. プリフェッチ

`<Link>` コンポーネントがビューポートに入った際に自動的に実行されます。

| ルートの種類 | プリフェッチの挙動 |
|---|---|
| 静的ルート（Static Route） | ルート全体がプリフェッチされる |
| 動的ルート（Dynamic Route） | 原則スキップ。`loading.tsx` が存在する場合のみ部分的にプリフェッチされる |

### 3. ストリーミング

サーバーがルート全体をレンダリングし終えるのを待たず、準備ができた部分から順次クライアントへ送信する機能です。

- **部分的なプリフェッチ**: 動的ルートにおいて、共有レイアウトやローディングスケルトンを事前に要求することが可能
- **実装**: `loading.tsx` をルートフォルダに配置することで、Next.jsは自動的にページコンテンツを `<Suspense>` 境界でラップしフォールバックUIを表示する

### 4. クライアントサイド遷移

`<Link>` コンポーネントによる遷移はページ全体の再読み込みを発生させません。

- 共有UIの状態やスクロール位置を維持する
- 現在のページをプリフェッチされたローディング状態、または新しいページコンテンツで動的に置き換える

## パフォーマンス阻害要因と最適化手法

### 動的ルートの改善

**`loading.tsx` の欠如**

動的ルートへ移動する際、サーバーの応答を待つ間アプリが反応していないように見えることがあります。`loading.tsx` を追加することで即座に視覚的なフィードバックを返し、インタラクティブな状態を維持できます。

**`generateStaticParams` の活用**

本来プリレンダリング可能な動的セグメントでこれが欠落していると、リクエスト時のダイナミックレンダリングにフォールバックします。

```ts
export async function generateStaticParams() {
  const hotels = await db.hotels.findMany()
  return hotels.map(hotel => ({ id: hotel.id }))
}
```

### ネットワークとハイドレーションの最適化

**低速ネットワークへの対応**

プリフェッチが完了する前にリンクがクリックされた場合、`useLinkStatus` フックを使用して進行中の遷移に対する即時フィードバック（プログレスバーなど）を表示することが有効です。

**ハイドレーション遅延の解消**

`<Link>` はClient Componentであり、プリフェッチ開始にはハイドレーションが必要です。大きなJSバンドルはこれを遅らせるため、以下が推奨されます。

- `@next/bundle-analyzer` による依存関係の削減
- ロジックをサーバー側へ移動させる

### プリフェッチのカスタマイズ

**無効化**

```tsx
// クリック時のみフェッチ（大量のリンクがある場合に有用）
<Link href="/page" prefetch={false}>...</Link>
```

**ホバー時プリフェッチ**

ビューポート入時ではなく、ホバーしたタイミングでプリフェッチを行うことでリソース使用量を抑えつつ、訪問の可能性が高いルートの準備を整えられます。

```tsx
'use client'
import Link from 'next/link'
import { useState } from 'react'

function HoverPrefetchLink({ href, children }) {
  const [active, setActive] = useState(false)

  return (
    <Link
      href={href}
      prefetch={active ? null : false}
      onMouseEnter={() => setActive(true)}
    >
      {children}
    </Link>
  )
}
```

## 補足：ネイティブHistory APIとの統合

Next.jsはブラウザ標準の `window.history.pushState` および `window.history.replaceState` をサポートしており、ページのリロードなしで履歴スタックを更新できます。

| メソッド | 挙動 | ユースケース |
|---|---|---|
| `pushState` | 履歴スタックに新しいエントリを追加 | 商品のソート状態の反映など |
| `replaceState` | 現在の履歴エントリを置き換え | アプリケーションの言語切り替えなど |

これらのメソッドはNext.jsルーターと統合されているため、`usePathname` や `useSearchParams` との同期が自動的に行われます。
