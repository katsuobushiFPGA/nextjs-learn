# Next.js 16 デプロイ戦略まとめ

## 概要

デプロイ先の選択がアプリケーションの機能的限界を決める。Next.js 16 では Node.js サーバー、Docker、静的エクスポートの選択肢があるが、インフラ選定はアーキテクチャ上の負債に直結する。

---

## Fact 1：Docker はローカル開発に最適ではない

- Docker は**本番環境のポータビリティ・再現性**において最強の選択肢
- ただし Mac / Windows のローカル開発では、仮想化レイヤーによる **I/O パフォーマンスの低下**が発生する
- ローカルでは `npm run dev` によるネイティブ実行を推奨
  - Next.js の Fast Refresh や高速な開発サイクルを最大限に活かせる

> **使い分け**：ローカル → `npm run dev` ／ 本番 → Docker

---

## Fact 2：デプロイ先で利用可能な機能が二分される

| デプロイ先 | 機能サポート |
|---|---|
| Node.js サーバー / Docker | **全機能利用可能**。サーバーサイド動的ランタイムをフル活用 |
| 静的エクスポート | **機能制限あり** |

- フルスタック機能を活用するなら Node.js サーバー / Docker が必須
- 対応プロバイダー例：Railway、Flightcontrol、Google Cloud Run
- 静的エクスポートを安易に選ぶと、後からサーバーサイド機能が必要になった際に**大きなマイグレーションコスト**が発生する

---

## Fact 3：静的エクスポートは「制限」ではなく「戦略的出発点」

- SPA としてスタートし、必要に応じてサーバー機能を追加していく**段階的な拡張モデル**として有効
- 初期フェーズのホスティング例：AWS S3、Nginx、Apache などの既存サーバーで低コストスタート
- ビジネス成長に合わせて Node.js サーバーサイド機能を追加していく設計が可能

---

## Fact 4：Deployment Adapters API によるベンダーロックイン解消（開発中）

- 特定プラットフォーム（Vercel、AWS、Cloudflare、Netlify 等）に依存しない**真のポータビリティ**を実現する標準 API
- 開発者が独自のアダプターを作成できるようになる
- 企業にとっての**ベンダーロックインリスクを解消**し、インフラ選択の自由度を大幅に向上させる

> 現在開発中。完了後、カスタムアダプター作成のドキュメントが追加予定。

---

## まとめ：デプロイ戦略の設計指針

| 観点 | 方針 |
|---|---|
| 開発効率 | ローカルは `npm run dev`、本番は Docker |
| 機能フル活用 | Node.js サーバー / Docker を選択 |
| スモールスタート | 静的エクスポートから段階的に拡張 |
| 将来的な移植性 | Adapters API の動向を注視 |
