# Next.js 16 の画像最適化が Web 開発の常識を変える 3 つの理由

## 背景

画像の遅延読み込みによるレイアウトシフト（CLS）や読み込み速度の低下は、ユーザーのストレスに留まらず、離脱率の上昇・Lighthouse スコアの低下・ビジネス成果への悪影響に直結する。

Next.js の `<Image>` コンポーネントは `<img>` タグの代替品ではなく、最新の Web 標準を自動適用してパフォーマンスを最大化するための戦略的ツールである。

---

## 理由 1：CLS の自動排除による究極の視覚的安定性

`public/` フォルダの画像を静的インポートした場合、Next.js がビルド時に `width` と `height` を自動決定する。これらの値はアスペクト比の確定に使われ、画像読み込み中の CLS を防止する。

**効果：** デザイナーから受け取った画像のサイズを手動で計算し CSS で領域確保する作業が不要になり、本来の実装に集中できる。

---

## 理由 2：自動最適化と心理的プレースホルダーによる DX・UX の両立

| 機能 | 概要 |
|------|------|
| **オンデマンド・リサイズ** | WebP 等の次世代フォーマットへの変換とデバイス別リサイズをサーバーサイドでリアルタイム実行。事前に複数解像度を用意する手間が不要 |
| **ネイティブ遅延読み込み** | ビューポートに入る直前まで読み込みを待機し、初期ロードのデータ通信量を最小化 |
| **ぼかしプレースホルダー** | `placeholder="blur"` により、真っ白な空白の代わりに低解像度のぼかし画像を先行表示。「コンテンツが準備されている」という安心感を与え、心理的な待ち時間を短縮して離脱を防ぐ |

---

## 理由 3：リモート画像への鉄壁のセキュリティ

外部サーバーやクラウドストレージ（AWS S3 等）の画像を扱う際は、`next.config.ts` で許可するドメインとパスをホワイトリスト形式で明示する必要がある。

```ts
// next.config.ts
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 's3.amazonaws.com',
        port: '',
        pathname: '/my-bucket/**', // パスを絞り込みセキュリティリスクを最小化
      },
    ],
  },
}

export default nextConfig
```

### リモート画像の技術的制約

Next.js はビルド中にリモートファイルにアクセスできないため、`width` / `height` の手動指定または `fill` プロパティの使用が必要。

この「手動設定の強制」は、ビルド時にサイズ不明なリモート画像であっても事前にサイズを宣言させることで CLS を確実に防ぐための、意図的な設計である。

---

## 結論

| 特徴 | メリット |
|------|----------|
| 自動サイズ決定 | ビジュアルの安定 |
| オンデマンド処理 | 圧倒的な配信スピード |
| 厳格なホワイトリスト | 鉄壁のセキュリティ |

これらが標準で組み込まれている現代において、旧来の `<img>` タグを使い続けることはパフォーマンスの機会損失につながる。フロントエンドの品質向上は、最も影響力の大きい「画像」の改善から始まる。
