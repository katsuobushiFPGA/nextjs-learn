# Next.js の「当たり前」が変わる。Server と Client コンポーネントを使いこなすための 5 つの決定的ポイント

## 1. イントロダクション：React 開発の新しい「標準」への招待

Next.js（App Router）の登場により、開発スタイルは根本的な再定義を迫られた。最大の転換点は、コンポーネントが「デフォルトで Server コンポーネントである」という点に集約される。

多くの開発者が「どちらを使えばいいのか？」という迷いを抱くが、この設計思想の本質は、パフォーマンスを「開発者の努力」から「フレームワークの構造」へと昇華させる**引き算の美学（Philosophy of Less）**にある。優れた First Contentful Paint（FCP）と開発効率を両立させるための新しい「標準」を紐解いていく。

---

## 2. ポイント 1：なぜ「サーバーがデフォルト」なのか？——パフォーマンスへの強制力

Next.js が Server コンポーネントをデフォルトに据えたのは、それが最も合理的かつ強力な「パフォーマンス・ガードレール」になるからだ。

これまで、JavaScript のバンドルサイズ削減は、エンジニアがコード分割や Tree Shaking を駆使して戦う、いわば終わりのない「バンドルサイズ戦争」だった。しかし Server コンポーネントはこの戦いの前提を変える。**サーバー側で実行されるコードはブラウザに送信されない**ため、どれほど重厚なライブラリを使用しても、クライアントの JS 負荷は増えない。

Server コンポーネントが役立つ場面は以下の通りだ。

- データベースや API からソースの近くでデータをフェッチする
- API キーやトークンなどのシークレット情報をクライアントに公開せずに使用する
- ブラウザに送信される JavaScript の量を削減する

この設計思想の転換により、開発者は「いかに JS を削るか」という後追いの最適化から解放され、構造的に軽量なアプリケーションを構築できるようになる。

---

## 3. ポイント 2：`'use client'` はコンポーネントごとの宣言ではない——「境界線」の真実

`'use client'` ディレクティブを「クライアント用コンポーネントにするためのフラグ」と捉えるのは不十分だ。正しくは、Server モジュールグラフと Client モジュールグラフの**「境界線（Boundary）」**を定義する宣言である。

一度この境界を引けば、その配下でインポートされるすべてのモジュールや子コンポーネントは自動的にクライアントバンドルに含まれる。すべてのファイルに記述する必要がないのは、この「境界」の概念があるからだ。

また、注目すべきは初回ロード時とその後の挙動の違いだ。

| タイミング | 挙動 |
|---|---|
| 初回ロード時 | ユーザー体験を損なわないよう、サーバー側で Client コンポーネントも HTML として事前レンダリングされる |
| その後のナビゲーション | サーバーでレンダリングされた HTML を介さず、クライアント側で完全にレンダリングが行われる |

この動的な切り替えを理解してこそ、真の境界設計が可能になる。

---

## 4. ポイント 3：逆転の発想——Client コンポーネントの中に Server コンポーネントを「埋め込む」

「クライアント側の中にサーバー側は置けない」という思い込みは、柔軟な設計を阻害する。`children` や props として渡す**「インターリーブ（挟み込み）」パターン**を活用しよう。

例えば、状態管理が必要な `Modal`（Client）の中に、サーバー側でデータを取得する `Cart`（Server）を表示する構成がこれに当たる。これが可能なのは、Server コンポーネントがクライアントに届く前に「レンダリング済みの UI」として確定しているからだ。

ただし、絶対に忘れてはならないのが**「シリアライズ（Serializable）」の制約**だ。Server から Client へ渡す props は、関数やクラスインスタンスではなく、JSON 化可能なデータである必要がある。

この構成は、サーバー上で生成された RSC Payload 内に「Client コンポーネントを配置する場所の参照（プレースホルダー）」が含まれる仕組みによって実現されている。

---

## 5. ポイント 4：RSC Payload——ブラウザが受け取る「第 3 のデータ形式」

Next.js の魔法の正体は、HTML と JavaScript の間に存在する**「React Server Component Payload（RSC Payload）」**というコンパクトなバイナリ表現にある。

RSC Payload には以下の情報が凝縮されている。

- Server コンポーネントのレンダリング結果
- Client コンポーネントの配置場所を示すプレースホルダーと JS ファイルへの参照
- Server から Client へ渡されるシリアライズされた props

これが「高速なプレビュー」と「インタラクティブ性」を両立させる鍵だ。クライアント側の React はこのペイロードを読み解き、DOM を更新しながら、適切なタイミングで Client コンポーネントのハイドレーション（イベントハンドラーのアタッチ）をオーケストレーションする。

---

## 6. ポイント 5：うっかりミスを防ぐ `server-only` パッケージの守護

技術的な疎かさは、時に重大なセキュリティリスクを招く。例えば `lib/data.ts` 内に定義された `getData()` 関数が機密性の高い `API_KEY` を参照している場合、これを誤って Client コンポーネントにインポートしてしまうのは致命的だ。

これを未然に防ぐのが `server-only` パッケージだ。このモジュールをインポートしておけば、誤った環境でのロードを**ビルド時にエラーとして検知**できる。実行時ではなくビルド時に防ぐという点にエンジニアリングとしての価値がある。

```ts
import 'server-only'  // 誤って Client からインポートされたらビルドエラー

export async function getData() {
  const res = await fetch('https://api.example.com', {
    headers: { authorization: process.env.API_KEY },
  })
  return res.json()
}
```

また、環境変数のプレフィックスルールもこの安全装置の一翼を担っている。

| 変数名 | クライアントへの公開 |
|---|---|
| `NEXT_PUBLIC_KEY` | クライアントバンドルに含まれる（意図的に公開） |
| `API_KEY` | 空文字列に置き換えられる（漏洩しない） |

「プレフィックスがないものは空文字列に置き換える」という挙動は、意図しない情報漏洩を物理的に遮断する Next.js の強力なセキュリティ思想の現れだ。

---

## 結論：これからの React コンポーネント設計に求められる視点

Server と Client コンポーネントの使い分けは、どちらかを選ぶ「トレードオフ」ではなく、それぞれの環境の強みを最大化する「共存」の設計だ。**サーバーはデータとセキュリティを、クライアントはユーザーの感性に触れるインタラクションを受け持つ。** この役割分担を意識することで、コードの責務はより明確になる。

最後に、あなたのプロジェクトを見直してみてほしい。

> 「あなたの現在のプロジェクトで、本当は Client コンポーネントである必要がない（単なるデータ表示に過ぎない）部分はどこですか？」

その問いに答えることが、次世代のパフォーマンス・スタンダードへと繋がるはずだ。
