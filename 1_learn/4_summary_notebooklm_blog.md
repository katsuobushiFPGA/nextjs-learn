# Next.jsの「爆速」ナビゲーションを支える5つの驚きの仕組み：ユーザーを待たせない設計術

## 1. イントロダクション

ウェブサイトでリンクをクリックした際、画面が白く反転したり、再読み込みのために一瞬操作が止まったりすることにストレスを感じたことはないでしょうか。ユーザーにとって、クリックに対する反応の遅れは、アプリケーションの信頼性や品質を疑わせる致命的な「違和感」へと直結します。

Next.jsは、デフォルトでサーバーレンダリングを採用していながら、まるでSPAのような極めて滑らかな操作感を提供します。本記事では、ユーザーを1ミリ秒も待たせないための「5つの驚きの仕組み」を深掘りします。

---

## 2. 【驚き1】視界に入るだけで準備開始：`<Link>` コンポーネントの自動プリフェッチ

Next.jsにおける高速化の第一歩は、ユーザーがアクションを起こす「前」に始まっています。`<Link>` コンポーネントは、ブラウザのビューポートに入ったリンク先を自動的に検出し、バックグラウンドでそのルートの読み込みを開始します。

### ルートの種類によるプリフェッチの違い

| ルートの種類 | プリフェッチの挙動 |
|---|---|
| 静的ルート（Static Route） | ルート全体がプリフェッチされる |
| 動的ルート（Dynamic Route） | デフォルトはスキップ。`loading.tsx` が存在する場合のみ部分的プリフェッチが実行される |

ユーザーがクリックを「決断」し、マウスを動かしている数ミリ秒の間に、必要なデータはすでに手元に届いています。この先回りこそが「待ち時間ゼロ」の体験を生み出します。

> *"Prefetching is the process of loading a route in the background before the user navigates to it. This makes navigation between routes in your application feel instant..."*

---

## 3. 【驚き2】「全部待つ」必要はない：Streamingと部分的プリフェッチ

動的ルートにおいて「サーバーがすべてのデータを準備するまで待つ」という旧来の常識は、Next.jsの「Streaming」によって過去のものとなりました。

`loading.tsx` を用意することで、Next.jsはルートの「共有レイアウト」と「ローディングスケルトン」を事前にクライアントへ送り届けることが可能になります。

```
クリック
  ↓
レイアウト＋スケルトンを即表示（プリフェッチ済み）
  ↓（バックグラウンドでデータ取得）
コンテンツに差し替え
```

全体のレンダリング完了を待つのではなく、「まず動く部分・見える部分を即座に出す」というアプローチがユーザーの体感速度（Perceived Performance）を劇的に向上させます。

> `loading.tsx` を定義しないことは、この強力な「部分的プリフェッチ」の恩恵を自ら放棄しているのと同義です。

---

## 4. 【驚き3】サーバーサイドなのにSPAの操作感：クライアントサイド・トランジション

従来のSSRはページ遷移のたびにフルリロードされ、アプリケーションの状態（State）が消失し、スクロール位置もリセットされていました。Next.jsはこの問題を「クライアントサイド・トランジション」で解決しています。

### セグメントベースの差し替え

Next.jsはページ全体を書き換えるのではなく、変更が必要なルートセグメントのみをサーバーから取得してクライアント側で動的に差し替えます。これにより共有レイアウト内の状態を維持したまま、コンテンツのみを滑らかに更新できます。

| | 従来のSSR | Next.js |
|---|---|---|
| ページ遷移 | フルリロード | セグメントのみ差し替え |
| 状態（State） | 消失 | 維持 |
| スクロール位置 | リセット | 制御可能 |

> *"Client-side transitions are what makes a server-rendered apps feel like client-rendered apps."*

---

## 5. 【驚き4】「遅い」と感じる原因は設計にあり：動的ルートの最適化とハイドレーション

### generateStaticParams の失念

ビルド時にレンダリング可能なルートが `generateStaticParams` を欠いているためにリクエスト時の動的レンダリングへフォールバックする現象は、パフォーマンスにおける「サイレントキラー」です。

```ts
export async function generateStaticParams() {
  const hotels = await db.hotels.findMany()
  return hotels.map(hotel => ({ id: hotel.id }))
}
```

### ハイドレーションの遅延

`<Link>` はClient Componentであり、ハイドレーションが完了するまでプリフェッチを開始できません。

```
巨大なJSバンドル
　↓
ハイドレーションが遅れる
　↓
プリフェッチが発火しない
　↓
「待ち時間ゼロ」の恩恵を受けられない
```

**対策:**
- `@next/bundle-analyzer` でバンドルサイズを削ぎ落とす
- ロジックを極力サーバーサイドへ寄せる

---

## 6. 【驚き5】低速ネットワークさえも味方につける：useLinkStatusによるフィードバック

ネットワークが不安定な状況ではプリフェッチが間に合わないことも想定しなければなりません。`useLinkStatus` フックを利用すれば、遷移が進行中であることを即座に視覚化できます。

### チラつき（Flickering）防止のデバウンス

微細な遅延のたびにインジケーターを出すと逆に画面がチラついて体験を損ねます。**「100msのデバウンス」**を挟むのが定石です。

```ts
// 100ms以上かかった場合のみ表示する
const indicatorStyle = {
  opacity: pending ? 1 : 0,
  transition: 'opacity 300ms ease-in 100ms'
}
```

---

## 7. 結論：ナビゲーションは「体験」そのものである

Next.jsが提供するナビゲーション体験は、単なる機能の羅列ではありません。

| 仕組み | 役割 |
|---|---|
| 自動プリフェッチ | クリック前にデータを先読み |
| Streaming | 準備できた部分から即座に表示 |
| セグメントベースの更新 | 必要な部分だけ差し替え |
| フィードバック制御 | 低速環境でもユーザーを不安にさせない |

これらが有機的に結合することで、「ユーザーの思考を妨げないシームレスな流れ」が完成します。

`loading.tsx` の配置や静的パラメータの最適化を「忘れる」という選択肢はありません。それは、ユーザーへのホスピタリティを欠くことと同じだからです。

> あなたのアプリケーションは、ユーザーの「クリック」という意思表示に対して、0ミリ秒で応えられていますか？
