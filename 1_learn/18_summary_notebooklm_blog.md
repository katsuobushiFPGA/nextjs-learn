# Next.js 16 新機能・アップグレードまとめ

## 概要

Next.js 16 は「キャッシュ制御の複雑さ」「認証エラー処理のボイラープレート」といった長年の課題に対する解答として設計されており、開発体験（DX）を大幅に向上させる。

---

## アップグレード手法

### 自動アップグレード（推奨）

`upgrade codemod` CLI コマンドを使用。大規模プロジェクトにおける手動の依存関係解消は人的ミスを誘発するリスクが高く、自動化による「規約に基づいた安全な移行」が推奨される。

### 手動アップグレード

以下を最新版に更新する。

- `next`（Next.js 本体）
- `react`（Next.js 16 は React の最新機能と密接に連携するため、足並みを揃えて更新が必要）

---

## Canary で導入された主要機能

### キャッシング（`"use cache"` 系）

| API | 概要 |
|---|---|
| `"use cache"` | 関数・コンポーネント単位でキャッシュを有効にする新しいディレクティブ |
| `cacheLife` | コンポーネントレベルで詳細な TTL（有効期限）を設定 |
| `cacheTag` | データにタグを付与し、オンデマンドでの再検証（Revalidation）を実現 |
| `cacheComponents` | コンポーネントのレンダリング結果そのものをキャッシュ対象にする |

**設計上の意義**：これまでキャッシュ制御は `fetch` のオプションや `revalidatePath` などネットワークレイヤーに依存していた。`"use cache"` によりキャッシュはコンポーネント自身のビジネスロジックとして定義でき、「責務の分離」が実現される。

---

### 認証エラー制御（`authInterrupts` 系）

| API | 概要 |
|---|---|
| `forbidden` / `forbidden.js` | ログイン済みだが権限なし（403）の場合のエラー表示 |
| `unauthorized` / `unauthorized.js` | 未ログイン状態（401）を検知した際の表示 |
| `authInterrupts` | 認証状態に応じてレンダリングを中断し、適切なフローへ誘導 |

**設計上の意義**：従来は条件分岐や手動の `redirect()` をページコンポーネント内に記述する必要があり、可読性を下げていた。ファイルベースのコンベンション（`forbidden.js` 等）により、ページ本体から条件分岐のボイラープレートを排除でき、コンポーネントを純粋なビジネスロジックの記述に専念させられる。「**Convention over Configuration**」の精神が体現された進化といえる。

---

## まとめ

| 課題 | Next.js 16 の解答 |
|---|---|
| キャッシュ制御の複雑さ | `"use cache"` 系の宣言的 API によるコンポーネントレベルのキャッシュ管理 |
| 認証エラーのボイラープレート | ファイルベースの `forbidden.js` / `unauthorized.js` による規約ベースの実装 |
| アップグレードコスト | `upgrade codemod` による安全な自動移行 |
