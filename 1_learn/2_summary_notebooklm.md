# Next.js プロジェクト構造と組織化

## 概要

Next.js の App Router では、フォルダ構造が URL セグメントを定義し、特定のファイル名（`page.js`、`layout.js` など）が特別な役割を担います。

プロジェクトの組織化については **unopinionated（主観を押し付けない）** という姿勢を取っており、ファイルを `app` 内にコロケーションするか外で管理するかは開発者に委ねられています。

---

## 1. フォルダとファイルの規約

### 最上位レベルのフォルダ

| フォルダ | 用途 |
|---|---|
| `app` | App Router（推奨） |
| `pages` | Pages Router |
| `public` | 配信される静的資産 |
| `src` | 任意。アプリのソースコードを格納 |

### ルーティングファイル

`app` ディレクトリ内では以下のファイル名が予約されています。

| ファイル名 | 拡張子 | 用途 |
|---|---|---|
| `layout` | `.js` `.jsx` `.tsx` | 共通 UI。子セグメントをラップ |
| `page` | `.js` `.jsx` `.tsx` | ルートを公開するための固有 UI |
| `loading` | `.js` `.jsx` `.tsx` | Suspense ベースのローディング UI |
| `not-found` | `.js` `.jsx` `.tsx` | 404 エラー用 UI |
| `error` | `.js` `.jsx` `.tsx` | エラー境界 UI |
| `global-error` | `.js` `.jsx` `.tsx` | グローバルエラー境界 UI |
| `route` | `.js` `.ts` | API エンドポイント |
| `template` | `.js` `.jsx` `.tsx` | 再レンダリングされるレイアウト |
| `default` | `.js` `.jsx` `.tsx` | パラレルルートのフォールバック |

---

## 2. ルーティングの高度な機能

### 動的ルート

| 記法 | 説明 | 例 |
|---|---|---|
| `[segment]` | 単一パラメータ | `/blog/[slug]` |
| `[...segment]` | キャッチオール | `/shop/[...slug]` → `/shop/clothing/shirts` |
| `[[...segment]]` | オプショナル・キャッチオール | パラメータなしのルートにも一致 |

### 組織化のための特殊フォルダ

| 記法 | 説明 |
|---|---|
| `(folderName)` | ルートグループ。URL に影響せずルートをグループ化 |
| `_folderName` | プライベートフォルダ。ルーティングから除外 |
| `@slot` | パラレルルート。1 つのレイアウト内で複数ページを同時表示 |
| `(.)folder` `(..)folder` | インターセプトルート。別ルートをモーダル等で表示 |

---

## 3. コンポーネントのレンダリング順序

同一セグメント内での優先順位は以下の通りです。

1. `layout.js`
2. `template.js`
3. `error.js`
4. `loading.js`
5. `not-found.js`
6. `page.js` または ネストされた `layout.js`

---

## 4. プロジェクトの組織化戦略

`page.js` または `route.js` が存在するまでルートは公開されないため、コンポーネントやテストを同じフォルダに安全にコロケーションできます。

主な戦略は以下の 3 つです。

**① `app` 外に保存**
`app` ディレクトリをルーティング専用とし、共有コードはルート直下で管理。

```
your-project
├── app
├── components   # 共有コード
└── lib
```

**② `app` 内の最上位フォルダに保存**
アプリ全体で共有するコードを `app` 直下の共通フォルダで管理。

```
app
├── _components  # 共有コード
├── _lib
├── dashboard
└── page.js
```

**③ 機能またはルートごとに分割**
グローバルな共有コードは `app` 直下に、ルート固有のコードはそのセグメント内に配置。

```
app
├── dashboard
│   ├── _components  # dashboard 専用
│   └── page.js
└── page.js
```

## 6. 設計上のポイント

**複数ルートレイアウト**
最上位の `layout.js` を削除し、各ルートグループに独自の `layout.js` を配置することで、セクションごとに完全に独立した UI を持つアプリを構築できます。この場合、各レイアウトに `<html>` と `<body>` タグが必要です。

**プライベートフォルダの利点**
将来の Next.js のファイル規約との命名競合を防ぎ、エディタ上での管理も容易になります。

**`src` フォルダの活用**
`next.config.js` などの設定ファイルとアプリコードを分離したい場合、`app` ディレクトリを `src` 配下に移動できます。
